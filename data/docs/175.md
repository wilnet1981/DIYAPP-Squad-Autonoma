# DIYAPP Evolution - V12 Core - Documento de Arquitetura

## 1. DECISÃO DE ARQUITETURA PRINCIPAL

**ADR-001: Arquitetura V12 - Stack e Padrões Fundamentais**
**Data:** 2024-01-15
**Status:** Aceita
**Autores:** Tech Lead, Especialista Infra, Especialista Backend

### CONTEXTO:
O DIYAPP evoluiu para uma plataforma complexa com múltiplos módulos (WhatsApp Automation, CRM, E-commerce, Analytics). A arquitetura atual (monolito Node.js + MongoDB) apresenta:
1. Acoplamento excessivo entre módulos
2. Dificuldade de deploy independente
3. Escalabilidade limitada por recursos compartilhados
4. Dívida técnica acumulada em padrões inconsistentes

### DECISÃO:
Adotar arquitetura de microsserviços com API Gateway, utilizando:
- **Frontend:** React 18 com TypeScript + Vite
- **Backend:** Node.js 20 LTS + TypeScript
- **API Gateway:** Kong (open-source)
- **Banco de Dados:** PostgreSQL 16 (principal) + Redis 7 (cache/sessões)
- **Mensageria:** RabbitMQ 3.12
- **Containerização:** Docker + Docker Compose (dev) / Kubernetes (prod)
- **CI/CD:** GitHub Actions + ArgoCD

### OPÇÕES CONSIDERADAS:

**Opção A: Monolito Aprimorado**
- Prós: Simplicidade de deploy, debugging facilitado, transações ACID fáceis
- Contras: Escalabilidade limitada, acoplamento mantido, risco de "big ball of mud"
- **Rejeitada:** Não atende requisito de evolução independente dos módulos

**Opção B: Microsserviços com NestJS**
- Prós: Framework maduro, estrutura padronizada, ecossistema TypeScript
- Contras: Curva de aprendizado, overhead de abstrações
- **Rejeitada:** Overengineering para nossa escala atual

**Opção C: Microsserviços com Node.js puro + Fastify**
- Prós: Performance superior, leveza, flexibilidade total
- Contras: Necessidade de mais boilerplate, menos convenções
- **Escolhida:** Balance ideal entre performance e manutenibilidade

### CONSEQUÊNCIAS:
**Positivas:**
- Deploy independente por módulo
- Escalabilidade horizontal granular
- Stack moderna e suportada long-term
- Isolamento de falhas entre serviços

**Negativas:**
- Complexidade operacional aumentada
- Necessidade de orquestração (K8s)
- Transações distribuídas mais complexas
- Latência de rede entre serviços

**Riscos:**
1. **Complexidade excessiva:** Mitigar com documentação clara e onboarding
2. **Custos de infra:** Monitorar com CloudWatch/Prometheus
3. **Debugging distribuído:** Implementar OpenTelemetry + Jaeger

**REVISÃO:** 2024-07-15 (6 meses para avaliação)

## 2. PADRÕES DE CÓDIGO

### TypeScript
- Strict mode obrigatório
- Interfaces > Types para objetos
- Nomes de arquivos: kebab-case
- Nomes de tipos/interfaces: PascalCase
- Evitar `any` - usar `unknown` quando necessário

### Node.js/Backend
- Fastify como framework HTTP
- Zod para validação de schemas
- Prisma como ORM
- Winston para logging estruturado
- Arquitetura: Controller → Service → Repository

### React/Frontend
- Functional components com hooks
- TanStack Query para data fetching
- Zustand para gerenciamento de estado
- React Hook Form + Zod para forms
- Tailwind CSS para estilização

## 3. CONVENÇÕES DE COMMIT
```
<type>(<scope>): <subject>

<body>

<footer>
```
Tipos: feat, fix, docs, style, refactor, test, chore, perf, ci, build
Exemplo: `feat(auth): implement login with 2FA`

## 4. TESTES
- Cobertura mínima: 80%
- Backend: Jest + Supertest
- Frontend: Vitest + React Testing Library
- E2E: Playwright
- Testes devem ser independentes e idempotentes

## 5. SEGURANÇA
- Variáveis sensíveis no .env (nunca no código)
- Validação de input em todas as camadas
- Rate limiting no API Gateway
- CORS configurado por ambiente
- Headers de segurança (Helmet)

## 6. PERFORMANCE
- Cache com Redis para dados frequentemente acessados
- Paginação em todas as listas (> 20 itens)
- Compression no API Gateway
- Lazy loading no frontend
- Monitoring com Prometheus + Grafana

## 4. ESTRATÉGIA DE DEPLOY CONTÍNUO

```yaml
# .github/workflows/deploy.yml
name: Deploy to Kubernetes

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: [auth-service, whatsapp-service, crm-service, frontend]
    
    steps:
    - uses: actions/checkout@v3
    
    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '20'
        cache: 'npm'
        cache-dependency-path: ./services/${{ matrix.service }}/package-lock.json
    
    - name: Install dependencies
      run: npm ci
      working-directory: ./services/${{ matrix.service }}
    
    - name: Run tests
      run: npm test -- --coverage
      working-directory: ./services/${{ matrix.service }}
    
    - name: Upload coverage
      uses: codecov/codecov-action@v3
      with:
        file: ./services/${{ matrix.service }}/coverage/lcov.info

  build-and-push:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    
    steps:
    - uses: actions/checkout@v3
    
    - name: Set up Docker Buildx
      uses: docker/setup-buildx-action@v2
    
    - name: Login to DockerHub
      uses: docker/login-action@v2
      with:
        username: ${{ secrets.DOCKER_USERNAME }}
        password: ${{ secrets.DOCKER_PASSWORD }}
    
    - name: Build and push
      run: |
        SERVICES=("auth-service" "whatsapp-service" "crm-service" "frontend" "api-gateway")
        for SERVICE in "${SERVICES[@]}"; do
          docker build -t ${{ secrets.DOCKER_USERNAME }}/diyapp-$SERVICE:${{ github.sha }} ./services/$SERVICE
          docker push ${{ secrets.DOCKER_USERNAME }}/diyapp-$SERVICE:${{ github.sha }}
        done

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    
    steps:
    - uses: actions/checkout@v3
    
    - name: Deploy to Kubernetes
      uses: azure/k8s-deploy@v1
      with:
        namespace: diyapp-production
        manifests: ./infrastructure/k8s/
        images: |
          ${{ secrets.DOCKER_USERNAME }}/diyapp-auth-service:${{ github.sha }}
          ${{ secrets.DOCKER_USERNAME }}/diyapp-whatsapp-service:${{ github.sha }}
          ${{ secrets.DOCKER_USERNAME }}/diyapp-crm-service:${{ github.sha }}
          ${{ secrets.DOCKER_USERNAME }}/diyapp-frontend:${{ github.sha }}
          ${{ secrets.DOCKER_USERNAME }}/diyapp-api-gateway:${{ github.sha }}
        kubectl-version: 'latest'
```

// src/middleware/errorHandler.ts - Tratamento centralizado de erros
import { Request, Response, NextFunction } from 'express';
import { ZodError } from 'zod';
import { config } from '../config';

export class AppError extends Error {
  constructor(
    public message: string,
    public statusCode: number = 500,
    public code?: string,
    public details?: any
  ) {
    super(message);
    this.name = 'AppError';
  }
}

export const errorHandler = (
  error: Error,
  req: Request,
  res: Response,
  next: NextFunction
) => {
  // Log do erro estruturado
  console.error(JSON.stringify({
    level: 'error',
    correlation_id: req.headers['x-correlation-id'] || 'unknown',
    user_id: (req as any).user?.id || 'anonymous',
    endpoint: `${req.method} ${req.path}`,
    error: {
      name: error.name,
      message: error.message,
      stack: config.nodeEnv === 'development' ? error.stack : undefined
    },
    timestamp: new Date().toISOString()
  }));

  // Tratamento de erros Zod (validação)
  if (error instanceof ZodError) {
    return res.status(400).json({
      error: 'Validation Error',
      message: 'Invalid input data',
      details: error.errors.map(err => ({
        field: err.path.join('.'),
        message: err.message
      })),
      code: 'VALIDATION_ERROR'
    });
  }

  // Tratamento de AppError
  if (error instanceof AppError) {
    return res.status(error.statusCode).json({
      error: error.name,
      message: error.message,
      code: error.code,
      details: error.details
    });
  }

  // Erro genérico (NUNCA expõe stack trace em produção)
  const response: any = {
    error: 'Internal Server Error',
    message: 'An unexpected error occurred'
  };

  if (config.nodeEnv === 'development') {
    response.debug = {
      message: error.message,
      stack: error.stack
    };
  }

  return res.status(500).json(response);
};

import { Request, Response, NextFunction } from 'express';
import { v4 as uuidv4 } from 'uuid';

export const requestLogger = (req: Request, res: Response, next: NextFunction) => {
  const startTime = Date.now();
  const correlationId = req.headers['x-correlation-id'] as string || uuidv4();
  
  // Adiciona correlation ID ao request
  req.headers['x-correlation-id'] = correlationId;
  
  // Log da requisição de entrada
  console.log(JSON.stringify({
    level: 'info',
    correlation_id: correlationId,
    user_id: (req as any).user?.id || 'anonymous',
    method: req.method,
    endpoint: req.path,
    user_agent: req.get('user-agent'),
    ip: req.ip,
    timestamp: new Date().toISOString(),
    type: 'request_start'
  }));

  // Intercepta a resposta para log
  const originalSend = res.send;
  res.send = function(body: any) {
    const duration = Date.now() - startTime;
    
    // Log da resposta
    console.log(JSON.stringify({
      level: 'info',
      correlation_id: correlationId,
      user_id: (req as any).user?.id || 'anonymous',
      method: req.method,
      endpoint: req.path,
      status_code: res.statusCode,
      duration_ms: duration,
      timestamp: new Date().toISOString(),
      type: 'request_end'
    }));

    return originalSend.call(this, body);
  };

  next();
};

-- Function to validate task hierarchy (no circular references)
CREATE OR REPLACE FUNCTION validate_task_hierarchy()
RETURNS TRIGGER AS $$
DECLARE
    current_parent_id UUID := NEW.parent_task_id;
    visited UUID[] := ARRAY[NEW.id];
BEGIN
    -- Check for circular reference
    WHILE current_parent_id IS NOT NULL LOOP
        -- If we find the new task's ID in the parent chain, it's a circle
        IF current_parent_id = ANY(visited) THEN
            RAISE EXCEPTION 'Circular task hierarchy detected';
        END IF;
        
        -- Add to visited and get next parent
        visited := visited || current_parent_id;
        SELECT parent_task_id INTO current_parent_id 
        FROM tasks 
        WHERE id = current_parent_id;
    END LOOP;
    
    RETURN NEW;
END;
$$ language 'plpgsql';

CREATE TRIGGER validate_task_hierarchy_trigger BEFORE INSERT OR UPDATE ON tasks
    FOR EACH ROW EXECUTE FUNCTION validate_task_hierarchy();

-- Function to update project updated_at when tasks change

RETURNS TRIGGER AS $$
BEGIN
    UPDATE projects 
    SET updated_at = NOW() 
    WHERE id = COALESCE(NEW.project_id, OLD.project_id);
    RETURN NEW;
END;
$$ language 'plpgsql';

## Agentes Especializados

### Agent Manager
**Responsabilidade**: Gerenciar ciclo de vida dos agentes especializados
**Mandato**: Criar, monitorar, escalar e remover agentes conforme demanda

### Project Orchestrator
**Responsabilidade**: Orquestrar execução de projetos complexos
**Mandato**: Dividir projetos em tarefas, alocar recursos, garantir conclusão

### Task Executor
**Responsabilidade**: Executar tarefas individuais
**Mandato**: Processar tarefas, reportar progresso, lidar com falhas

### Monitoring Service
**Responsabilidade**: Monitorar saúde do sistema
**Mandato**: Coletar métricas, detectar anomalias, disparar alertas

## Fluxo de Dados

1. **Entrada**: Cliente envia requisição para API Gateway
2. **Validação**: Gateway valida autenticação e rate limit
3. **Roteamento**: Request é roteado para serviço apropriado
4. **Processamento**: Serviço executa lógica de negócio
5. **Persistência**: Dados são salvos em camada de persistência
6. **Cache**: Resultados frequentes são cacheados
7. **Resposta**: Response é formatada e retornada ao cliente
8. **Logging**: Toda transação é logada para auditoria

## Decisões Arquiteturais

### ADR-001: Escolha do Banco de Dados
**Data**: 2024-01-15
**Status**: Aceito
**Contexto**: Necessidade de transações ACID e suporte a JSON
**Decisão**: PostgreSQL como banco primário, Redis para cache
**Consequências**: Maior complexidade operacional, mas garantia de consistência

### ADR-002: Estratégia de Autenticação
**Data**: 2024-01-20
**Status**: Aceito
**Contexto**: Múltiplos tipos de clientes (usuários, serviços, agentes)
**Decisão**: JWT para usuários, API Keys para serviços, mTLS para agentes
**Consequências**: Flexibilidade com segurança em camadas

### ADR-003: Padrão de Comunicação
**Data**: 2024-01-25
**Status**: Aceito
**Contexto**: Comunicação entre serviços em arquitetura distribuída
**Decisão**: REST para APIs públicas, gRPC para comunicação interna
**Consequências**: Performance melhorada na comunicação interna
