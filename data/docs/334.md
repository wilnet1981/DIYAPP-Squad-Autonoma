# DIYAPP Evolution - V12 Core - Análise Técnica e Arquitetura

## 1. Análise Técnica da V11

```markdown
# ADR-001: Análise Técnica da DIYAPP V11

**Data:** 2024-01-15
**Status:** Aceita
**Autores:** Tech Lead

## CONTEXTO:
A DIYAPP V11 evoluiu como um monolito Node.js/Express com funcionalidades crescentes:
- Sistema de autenticação JWT
- CRUD de projetos DIY
- Upload de imagens/mídia
- Sistema de comentários
- Integração básica com WhatsApp Web
- Dashboard administrativo

**Problemas identificados:**
1. Acoplamento excessivo entre módulos
2. Dificuldade para escalar componentes individualmente
3. Deploy único causa downtime total
4. Testes lentos devido à base de código monolítica
5. Onboarding complexo para novos devs
6. Performance degradada em picos de uso específicos

**Métricas críticas:**
- Tempo médio de build: 8.2 minutos
- Cobertura de testes: 68%
- Tempo de startup em produção: 45 segundos
- SLA atual: 95.3% (abaixo do target de 99.9%)
- Débito técnico acumulado: 42 pontos (alto)

## DECISÃO:
Realizar análise completa da V11 antes de definir arquitetura V12, documentando:
1. Dependências críticas
2. Pontos de falha
3. Padrões de uso
4. Custos operacionais
5. Limitações técnicas

## OPÇÕES CONSIDERADAS:
- **Opção A:** Análise superficial e migração direta - Rápido mas arriscado
- **Opção B:** Análise profunda com métricas - Mais lento mas seguro
- **Opção escolhida:** B - Justificativa: Risco técnico alto requer análise completa

## CONSEQUÊNCIAS:
**Positivas:** Decisões arquiteturais baseadas em dados reais
**Negativas:** Atraso de 2-3 dias no início do desenvolvimento
**Riscos:** Análise pode revelar problemas maiores que o esperado
```

## 2. Arquitetura V12 - Microserviços vs Monolito Otimizado

```markdown
# ADR-002: Arquitetura DIYAPP V12

**Data:** 2024-01-15
**Status:** Aceita
**Autores:** Tech Lead + Especialista Infra

## CONTEXTO:
Baseado na análise V11, precisamos escolher entre:
1. **Microserviços:** Separação por domínio (auth, projects, media, notifications)
2. **Monolito Otimizado:** Arquitetura modular com separação clara de responsabilidades

**Forças em jogo:**
- Time pequeno (5 devs)
- Necessidade de deploy rápido e frequente
- Recursos limitados de infraestrutura
- Requisito de estabilidade 100% autônoma
- Complexidade de gerenciar múltiplos serviços

## DECISÃO:
Adotar **Arquitetura Híbrida (Monolito Modular com caminho para microserviços)**:

1. **Fase 1 (V12.0 - V12.3):** Monolito modular com:
   - Domínios separados em módulos independentes
   - APIs internas bem definidas
   - Banco de dados único com schemas separados
   - Comunicação via eventos internos

2. **Fase 2 (V12.4+):** Evolução para microserviços quando:
   - Volume justificar (10k+ usuários ativos)
   - Time crescer para 8+ devs
   - Necessidade de escalar componentes específicos

## OPÇÕES CONSIDERADAS:

### Opção A: Microserviços Completo
**Prós:**
- Escalabilidade granular
- Deploys independentes
- Tecnologias específicas por serviço
- Isolamento de falhas

**Contras:**
- Complexidade operacional alta
- Latência em comunicações
- Overhead de monitoramento
- Custo infra mais alto
- Dificuldade para time pequeno

### Opção B: Monolito Otimizado
**Prós:**
- Simplicidade operacional
- Debugging mais fácil
- Performance em chamadas locais
- Custo infra mais baixo
- Ideal para time pequeno

**Contras:**
- Escalabilidade vertical limitada
- Deploy tudo ou nada
- Acoplamento potencial

### Opção C: Híbrida (Escolhida)
**Prós:**
- Balanceia simplicidade e escalabilidade
- Caminho de migração claro
- Time pode aprender gradualmente
- Reduz risco técnico

**Contras:**
- Design mais complexo inicialmente
- Necessidade de disciplina arquitetural

## CONSEQUÊNCIAS:
**Positivas:**
- Time pode operar com eficiência atual
- Caminho claro para crescimento
- Redução de risco operacional
- Manutenção simplificada

**Negativas:**
- Design inicial mais complexo
- Necessidade de refatoração futura

**Riscos:**
- Time pode negligenciar separação de módulos
- Mitigação: Code reviews rigorosos e métricas de acoplamento

**REVISÃO:** 2024-04-15 (após 3 meses em produção)
```

## 3. Stack Tecnológica V12

```markdown
# ADR-003: Stack Tecnológica DIYAPP V12

**Data:** 2024-01-15
**Status:** Aceita
**Autores:** Tech Lead + Especialista Infra + Especialista LLM

## CONTEXTO:
Necessidade de stack moderna, produtiva e com bom ecossistema para:
- Desenvolvimento rápido
- Estabilidade em produção
- Integração com IA/LLM
- Escalabilidade futura
- Manutenibilidade

## DECISÃO:
Adotar stack **MERN com PostgreSQL**:

### Backend:
- **Runtime:** Node.js 18 LTS
- **Framework:** Express.js + TypeScript
- **ORM:** Prisma (para type safety e migrations)
- **Autenticação:** JWT + refresh tokens
- **Validação:** Zod
- **Logging:** Winston + Elasticsearch
- **Monitoramento:** OpenTelemetry + Prometheus

### Frontend:
- **Framework:** React 18 + TypeScript
- **Build:** Vite
- **Estado:** Zustand (leve) + React Query
- **UI:** Chakra UI + Tailwind CSS
- **Roteamento:** React Router v6
- **Testes:** Vitest + React Testing Library

### Banco de Dados:
- **Primário:** PostgreSQL 15 (RDS/Aurora)
- **Cache:** Redis (ElastiCache)
- **Fila:** BullMQ (Redis-based)

### Infraestrutura:
- **Container:** Docker
- **Orquestração:** Docker Compose (dev) / ECS (prod)
- **CI/CD:** GitHub Actions
- **Monitoring:** Grafana + Loki
- **Alertas:** PagerDuty integrado

### IA/LLM:
- **Framework:** LangChain.js
- **Modelos:** GPT-4 Turbo + Claude 3 (fallback)
- **Embeddings:** OpenAI text-embedding-3-small
- **Vector DB:** pgvector (extensão PostgreSQL)
- **Cache semântico:** Redis

## OPÇÕES CONSIDERADAS:

### Backend Options:
- **NestJS:** Muito boilerplate para nosso caso
- **Fastify:** Performance melhor mas ecossistema menor
- **Express + TS:** Equilíbrio perfeito para nós

### Frontend Options:
- **Next.js:** Overkill para nosso SPA
- **Vue 3:** Time mais familiar com React
- **React + Vite:** Performance excelente, simples

### Database Options:
- **MongoDB:** Flexível mas schemaless arriscado
- **PostgreSQL:** Relacional + JSON + pgvector = perfeito

## CONSEQUÊNCIAS:
**Positivas:**
- Stack madura e bem documentada
- Grande comunidade
- Type safety end-to-end
- Performance excelente
- Custo controlado

**Negativas:**
- Learning curve para Prisma
- PostgreSQL requer mais schema design

**Riscos:**
- Complexidade de setup inicial
- Mitigação: Templates e documentação detalhada
```

### .github/workflows/ci.yml
```yaml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_PASSWORD: test
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 5432:5432
      
      redis:
        image: redis:7-alpine
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        ports:
          - 6379:6379

    steps:
    - uses: actions/checkout@v3
    
    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18'
        cache: 'npm'
    
    - name: Install dependencies
      run: npm ci
      
    - name: Lint
      run: npm run lint
      
    - name: Type check
      run: npm run type-check
      
    - name: Run unit tests
      run: npm test -- --coverage
      
    - name: Run integration tests
      run: npm run test:integration
      env:
        DATABASE_URL: postgresql://postgres:test@localhost:5432/diyapp_test
        REDIS_URL: redis://localhost:6379
        
    - name: Upload coverage
      uses: codecov/codecov-action@v3
      
    - name: Build
      run: npm run build
      
    - name: Security scan
      uses: snyk/actions/node@master
      env:
        SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
      with:
        args: --severity-threshold=high

  docker-build:
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main'
    
    steps:
    - uses: actions/checkout@v3
    
    - name: Set up Docker Buildx
      uses: docker/setup-buildx-action@v2
      
    - name: Login to DockerHub
      uses: docker/login-action@v2
      with:
        username: ${{ secrets.DOCKER_USERNAME }}
        password: ${{ secrets.DOCKER_PASSWORD }}
        
    - name: Build and push backend
      uses: docker/build-push-action@v4
      with:
        context: .
        file: ./infrastructure/docker/Dockerfile.backend
        push: true
        tags: |
          ${{ secrets.DOCKER_USERNAME }}/diyapp-backend:latest
          ${{ secrets.DOCKER_USERNAME }}/diyapp-backend:${{ github.sha }}
          
    - name: Build and push frontend
      uses: docker/build-push-action@v4
      with:
        context: ./frontend
        file: ./infrastructure/docker/Dockerfile.frontend
        push: true
        tags: |
          ${{ secrets.DOCKER_USERNAME }}/diyapp-frontend:latest
          ${{ secrets.DOCKER_USERNAME }}/diyapp-frontend:${{ github.sha }}

### .github/workflows/cd.yml
```yaml
name: CD Pipeline

on:
  workflow_run:
    workflows: ["CI Pipeline"]
    branches: [main]
    types:
      - completed

jobs:
  deploy-staging:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    environment: staging
    
    steps:
    - uses: actions/checkout@v3
    
    - name: Configure AWS credentials
      uses: aws-actions/configure-aws-credentials@v2
      with:
        aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
        aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        aws-region: us-east-1
        
    - name: Deploy to ECS
      run: |
        cd infrastructure/terraform
        terraform init
        terraform apply -auto-approve -var="environment=staging"
        
    - name: Run migrations
      run: |
        aws ecs run-task \
          --cluster diyapp-staging \
          --task-definition diyapp-migrations \
          --launch-type FARGATE \
          --network-configuration "awsvpcConfiguration={subnets=[subnet-xxx],securityGroups=[sg-yyy]}"
          
    - name: Health check
      run: |
        curl -f https://staging.diyapp.com/health || exit 1
        
    - name: Notify Slack
      uses: 8398a7/action-slack@v3
      with:
        status: ${{ job.status }}
        author_name: Deployment Bot
      env:
        SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production
    
    steps:
    - name: Wait for manual approval
      uses: trstringer/manual-approval@v1
      with:
        secret: ${{ github.token }}
        approvers: ${{ vars.PRODUCTION_APPROVERS }}
        minimum-approvals: 2
        
    - name: Deploy to production
      run: |
        cd infrastructure/terraform
        terraform apply -auto-approve -var="environment=production"
        
    - name: Run canary deployment
      run: |
        # Implementação de canary com 10% de tráfego inicial
        aws ecs update-service \
          --cluster diyapp-production \
          --service diyapp-backend \
          --desired-count 10
          
        sleep 300  # Wait 5 minutes
        
        # Monitor metrics
        # If metrics OK, scale to 100%
        
    - name: Post-deployment verification
      run: |
        # Run smoke tests
        npm run test:smoke -- --url=https://app.diyapp.com
        
        # Check error rates
        # Check performance metrics
```

### Engineering Standards Document
```markdown
# DIYAPP Engineering Standards V12

## 1. Code Style
- **TypeScript:** Strict mode enabled
- **Naming:** camelCase (variables), PascalCase (classes), UPPER_CASE (constants)
- **Imports:** Grouped (external, internal, types), alphabetical within groups
- **Functions:** Max 20 lines, single responsibility
- **Files:** Max 400 lines

## 2. Git Workflow
- **Branching:** Git Flow (main, develop, feature/*, hotfix/*)
- **Commits:** Conventional Commits (feat:, fix:, chore:, docs:, test:)
- **PR Size:** Max 400 lines changed
- **Reviews:** 2 approvals required (1 must be Tech Lead)
- **Squash Merge:** Always squash feature branches

## 3. Testing
- **Coverage:** Minimum 80% (backend), 70% (frontend)
- **Unit Tests:** Isolated, no external dependencies
- **Integration Tests:** Test full workflows
- **E2E Tests:** Critical user journeys only
- **Mocks:** Use MSW for API mocking

## 4. API Design
- **RESTful:** Resource-based endpoints
- **Versioning:** URL versioning (/api/v1/*)

@app.middleware("http")
async def logging_middleware(request: Request, call_next):
    """Middleware para logging estruturado e métricas"""
    start_time = time.time()
    correlation_id = request.headers.get("X-Correlation-ID", "unknown")
    
    # Log da requisição
    logger.info(
        "Request started",
        extra={
            "correlation_id": correlation_id,
            "method": request.method,
            "url": str(request.url),
            "client_ip": request.client.host if request.client else None,
            "user_agent": request.headers.get("user-agent")
        }
    )
    
    try:
        response = await call_next(request)
        duration = time.time() - start_time
        
        # Log da resposta
        logger.info(
            "Request completed",
            extra={
                "correlation_id": correlation_id,
                "method": request.method,
                "url": str(request.url),
                "status_code": response.status_code,
                "duration_ms": round(duration * 1000, 2)
            }
        )
        
        # Adiciona headers de métricas
        response.headers["X-Request-Duration"] = str(round(duration * 1000, 2))
        response.headers["X-Correlation-ID"] = correlation_id
        
        return response
        
    except Exception as e:
        duration = time.time() - start_time
        logger.error(
            "Request failed",
            extra={
                "correlation_id": correlation_id,
                "method": request.method,
                "url": str(request.url),
                "error": str(e),
                "duration_ms": round(duration * 1000, 2)
            },
            exc_info=True
        )
        raise

### 2.2 Vault para Gerenciamento de Segredos

**src/config/vault.js**
```javascript
const { Vault } = require('node-vault');
const crypto = require('crypto');

class SecretVault {
  constructor() {
    this.vault = Vault({
      apiVersion: 'v1',
      endpoint: process.env.VAULT_URL || 'http://localhost:8200',
      token: process.env.VAULT_TOKEN
    });
    
    this.localCache = new Map();
    this.encryptionKey = process.env.ENCRYPTION_KEY;
  }

  async getSecret(path, key) {
    try {
      // Tenta obter do Vault
      const result = await this.vault.read(`secret/data/${path}`);
      return result.data.data[key];
    } catch (error) {
      // Fallback para ambiente local com criptografia
      const encrypted = process.env[key];
      if (!encrypted) throw new Error(`Secret ${key} not found`);
      
      return this.decryptLocal(encrypted);
    }
  }

  async setSecret(path, key, value) {
    const encrypted = this.encryptLocal(value);
    
    try {
      await this.vault.write(`secret/data/${path}`, {
        data: { [key]: value }
      });
    } catch (error) {
      // Armazena localmente criptografado
      process.env[key] = encrypted;
    }
    
    this.localCache.set(key, value);
  }

  encryptLocal(text) {
    const iv = crypto.randomBytes(16);
    const cipher = crypto.createCipheriv('aes-256-gcm', 
      Buffer.from(this.encryptionKey, 'hex'), iv);
    
    let encrypted = cipher.update(text, 'utf8', 'hex');
    encrypted += cipher.final('hex');
    
    const authTag = cipher.getAuthTag();
    
    return `${iv.toString('hex')}:${encrypted}:${authTag.toString('hex')}`;
  }

  decryptLocal(encryptedText) {
    const [ivHex, encrypted, authTagHex] = encryptedText.split(':');
    
    const iv = Buffer.from(ivHex, 'hex');
    const authTag = Buffer.from(authTagHex, 'hex');
    const decipher = crypto.createDecipheriv('aes-256-gcm',
      Buffer.from(this.encryptionKey, 'hex'), iv);
    
    decipher.setAuthTag(authTag);
    
    let decrypted = decipher.update(encrypted, 'hex', 'utf8');
    decrypted += decipher.final('utf8');
    
    return decrypted;
  }

  rotateKeys() {
    // Rotação automática de chaves a cada 90 dias
    const newKey = crypto.randomBytes(32).toString('hex');
    this.encryptionKey = newKey;
    
    // Re-encrypt todos os segredos em cache
    for (const [key, value] of this.localCache.entries()) {
      const encrypted = this.encryptLocal(value);
      process.env[key] = encrypted;
    }
    
    return newKey;
  }
}

module.exports = new SecretVault();

class RBACManager {
  constructor() {
    this.rbac = new RBAC(policies);
    this.permissionCache = new Map();
  }

  async checkPermission(userRole, action, resource, context = {}) {
    const cacheKey = `${userRole}:${action}:${resource}:${JSON.stringify(context)}`;
    
    if (this.permissionCache.has(cacheKey)) {
      return this.permissionCache.get(cacheKey);
    }

    try {
      const allowed = await this.rbac.can(userRole, action, resource, context);
      this.permissionCache.set(cacheKey, allowed);
      
      // Log de auditoria para tentativas de acesso
      if (!allowed) {
        await this.logAccessAttempt(userRole, action, resource, context, false);
      }
      
      return allowed;
    } catch (error) {
      await this.logAccessAttempt(userRole, action, resource, context, false, error);
      return false;
    }
  }

  async logAccessAttempt(role, action, resource, context, success, error = null) {
    const auditLog = {
      timestamp: new Date().toISOString(),
      role,
      action,
      resource,
      context,
      success,
      ip: context.ip || 'unknown',
      userAgent: context.userAgent || 'unknown',
      error: error ? error.message : null
    };

    // Envia para serviço de auditoria
    console.log('[AUDIT]', JSON.stringify(auditLog));
    
    // TODO: Integrar com serviço de logging centralizado
  }

  // Princípio do menor privilégio
  getLeastPrivilegeRole(requiredActions) {
    const roles = ['guest', 'user', 'moderator', 'admin'];
    
    for (const role of roles) {
      const canAll = requiredActions.every(action => 
        this.rbac.canSync(role, action)
      );
      
      if (canAll) {
        return role;
      }
    }

class RateLimiter {
  constructor() {
    this.redis = new Redis({
      host: process.env.REDIS_HOST || 'localhost',
      port: process.env.REDIS_PORT || 6379,
      password: process.env.REDIS_PASSWORD
    });

    this.limits = {
      // OWASP A2: Proteção contra brute force
      auth: rateLimit({
        store: new RedisStore({
          client: this.redis,
          prefix: 'rl:auth:'
        }),
        windowMs: 15 * 60 * 1000, // 15 minutos
        max: 5, // 5 tentativas por IP
        message: 'Too many login attempts, please try again after 15 minutes',
        skipSuccessfulRequests: true
      }),

      // OWASP A8: Proteção contra abuso de API
      api: rateLimit({
        store: new RedisStore({
          client: this.redis,
          prefix: 'rl:api:'
        }),
        windowMs: 60 * 1000, // 1 minuto
        max: 100, // 100 requests por minuto
        message: 'Too many requests, please slow down',
        standardHeaders: true,
        legacyHeaders: false
      }),

      // Proteção específica por endpoint
      sensitive: rateLimit({
        store: new RedisStore({
          client: this.redis,
          prefix: 'rl:sensitive:'
        }),
        windowMs: 60 * 60 * 1000, // 1 hora
        max: 10, // 10 requests por hora
        message: 'Rate limit exceeded for sensitive operations',
        keyGenerator: (req) => `${req.ip}:${req.user?.id || 'anonymous'}`
      })
    };
  }

  getMiddleware(type) {
    return this.limits[type] || this.limits.api;
  }

  async resetLimit(key) {
    await this.redis.del(`rl:auth:${key}`);
    await this.redis.del(`rl:api:${key}`);
    await this.redis.del(`rl:sensitive:${key}`);
  }

  async getRemainingAttempts(ip, endpoint) {
    const key = `rl:${endpoint}:${ip}`;
    const attempts = await this.redis.get(key);
    const max = endpoint === 'auth' ? 5 : endpoint === 'sensitive' ? 10 : 100;
    
    return {
      remaining: max - (parseInt(attempts) || 0),
      max,
      reset: await this.redis.ttl(key)
    };
  }
}

module.exports = new RateLimiter();

        this.withdrawn_at = data.withdrawn_at;
        this.ip_address = data.ip_address;
        this.user_agent = data.user_agent;
        this.metadata = data.metadata ? JSON.stringify(data.metadata) : null;
        this.created_at = data.created_at || new Date().toISOString();
        this.updated_at = data.updated_at || new Date().toISOString();
    }

    static fromDB(row) {
        return new Consent({

        });

            id: this.id,
            user_id: this.user_id,
            consent_type: this.consent_type,

            version: this.version,

            withdrawn_at: this.withdrawn_at,

            user_agent: this.user_agent,
            metadata: this.metadata,
            created_at: this.created_at,
            updated_at: this.updated_at
        };

    isValid() {

# Referência da API DIYAPP V12

**Última atualização:** 2024-01-15  
**Versão da API:** 12.0.0  
**Owner:** Tech Writer - Squad Autônoma

## Visão Geral

A API DIYAPP V12 permite integração completa com o sistema DIYAPP Evolution. Todas as requisições devem usar HTTPS e incluir o token de autenticação no header.

## Autenticação

### Fluxo de Autenticação JWT

1. **Login** → Obtém access token e refresh token
2. **Usar access token** → Incluir no header `Authorization: Bearer <token>`
3. **Token expirado** → Usar refresh token para obter novo access token
4. **Logout** → Invalidar refresh token no cliente

### Headers de Autenticação

```http
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Content-Type: application/json
```

```javascript
// Exemplo usando fetch
const response = await fetch('https://api.diyapp.com/v12/auth/login', {

  headers: {
    'Content-Type': 'application/json',

    email: 'usuario@exemplo.com',

  })

// Serviço de integração com WhatsApp Business API
// Usando a biblioteca Baileys para prototipagem

const { default: makeWASocket, useMultiFileAuthState, DisconnectReason } = require('@whiskeysockets/baileys');

class WhatsAppBusinessService {
    constructor() {
        this.socket = null;
        this.isConnected = false;
        this.sessionPath = './whatsapp-session';
    }

    async initialize() {
        try {
            const { state, saveCreds } = await useMultiFileAuthState(this.sessionPath);
            
            this.socket = makeWASocket({
                auth: state,
                printQRInTerminal: true,
                defaultQueryTimeoutMs: 60_000
            });

            this.socket.ev.on('connection.update', (update) => {
                const { connection, lastDisconnect } = update;
                
                if (connection === 'close') {
                    const shouldReconnect = lastDisconnect?.error?.output?.statusCode !== DisconnectReason.loggedOut;
                    
                    if (shouldReconnect) {
                        this.initialize();
                    } else {
                        this.isConnected = false;
                        console.log('WhatsApp desconectado');
                    }
                } else if (connection === 'open') {
                    this.isConnected = true;
                    console.log('WhatsApp conectado com sucesso!');
                }
            });

            this.socket.ev.on('creds.update', saveCreds);

            return { success: true, qrCode: true };
        } catch (error) {
            console.error('Erro ao inicializar WhatsApp:', error);
            return { success: false,
