# DIYAPP Evolution - V12 Core - Backlog Refinamento

## Definition of Done Atualizado (V12)

### Condições Universais para Toda Entrega
1. **✓ Critérios de Aceite**: Todos os cenários Gherkin da história atendidos e validados pelo PO
2. **✓ Testes Automatizados**: Cobertura ≥ 85% para código novo (aumentado de 80% na V11)
3. **✓ Qualidade de Código**: Zero warnings críticos no linter (ESLint config rigoroso)
4. **✓ Revisão Técnica**: PR aprovado pelo Tech Lead + pelo menos um dev sênior
5. **✓ Documentação**: Tech Writer validou documentação de APIs/fluxos afetados
6. **✓ Segurança**: Especialista de Segurança validou (quando toca autenticação/dados)
7. **✓ Performance**: Resposta em < 200ms P95 para endpoints críticos
8. **✓ Autonomia**: Sistema deve operar 24h sem intervenção humana após deploy
9. **✓ Monitoramento**: Métricas essenciais instrumentadas (Prometheus/Grafana)
10. **✓ Rollback**: Plano de rollback documentado e testado para mudanças críticas

## Backlog Refinado - Sprint V12

### EPIC: Sistema de Autonomia Total
**Objetivo**: DIYAPP deve operar 100% autônomo por 30 dias consecutivos

#### História 1: Sistema de Auto-healing para Microsserviços
**HISTÓRIA**:
Como SRE,
Quero que o sistema detecte e recupere automaticamente falhas em microsserviços,
Para que a disponibilidade permaneça >99.9% sem intervenção manual.

**CRITÉRIOS DE ACEITE (Gherkin)**:
```
Cenário 1 — Detecção de falha de saúde
Dado que um microsserviço está respondendo com HTTP 500 por mais de 30 segundos
Quando o monitor de saúde executa sua verificação a cada 10 segundos
Então o sistema deve registrar a falha no log estruturado com severity=ERROR
E deve disparar alerta nível 2 no canal #system-alerts

Cenário 2 — Recuperação automática (reinício)
Dado que um microsserviço falhou 3 verificações consecutivas
Quando o sistema identifica que é um serviço stateless
Então o orquestrador deve reiniciar o container automaticamente
E deve verificar saúde após 60 segundos
E se saudável, registrar evento de recuperação bem-sucedida

Cenário 3 — Falha persistente (escalação)
Dado que um microsserviço falhou após 3 reinícios consecutivos
Quando o sistema detecta padrão de falha persistente
Então deve escalar para o Tech Lead via webhook
E deve provisionar instância substituta em outra zona de disponibilidade
E deve redirecionar tráfego gradualmente (blue-green)
```

**DEPENDÊNCIAS**: 
- História 3 (Sistema de Monitoramento Unificado)
- História 5 (Orquestração de Containers)

**MOCKUP/REFERÊNCIA**: N/A (backend)

**NOTAS TÉCNICAS**:
- Implementar usando Kubernetes Liveness/Readiness Probes
- Configurar backoff exponencial para reinícios
- Considerar implementar sidecar pattern para observabilidade

#### História 2: Sistema de Escalonamento Automático Horizontal
**HISTÓRIA**:
Como Product Manager,
Quero que o sistema escale horizontalmente baseado em métricas de carga,
Para que performance permaneça consistente durante picos de tráfego.

**CRITÉRIOS DE ACEITE (Gherkin)**:
```
Cenário 1 — Scale-out por CPU
Dado que a CPU média do cluster excede 70% por 2 minutos consecutivos
Quando o HPA (Horizontal Pod Autoscaler) verifica métricas
Então o sistema deve adicionar 1 réplica ao deployment afetado
E deve aguardar 3 minutos antes de nova decisão de escala

Cenário 2 — Scale-out por latência
Dado que o P95 de latência do serviço de API excede 300ms por 1 minuto
Quando o sistema monitora métricas de performance
Então deve aumentar réplicas em 50% (mínimo 1)
E deve redistribuir carga via service mesh

Cenário 3 — Scale-in seguro
Dado que a CPU média está abaixo de 30% por 5 minutos consecutivos
Quando o sistema verifica utilização de recursos
Então deve remover 1 réplica (nunca abaixo do mínimo configurado)
E deve drenar conexões graceful antes de terminar pod
E deve validar que métricas não degradam após scale-in
```

**DEPENDÊNCIAS**:
- História 1 (Auto-healing)
- História 6 (Service Mesh)

**MOCKUP/REFERÊNCIA**: N/A

**NOTAS TÉCNICAS**:
- Configurar cooldown periods para evitar oscilação
- Implementar pod disruption budgets para garantir disponibilidade
- Testar com carga sintética antes de produção

#### História 3: Sistema de Monitoramento Unificado com IA
**HISTÓRIA**:
Como Tech Lead,
Quero um dashboard unificado que detecte anomalias usando machine learning,
Para que possamos prever problemas antes que afetem usuários.

**CRITÉRIOS DE ACEITE (Gherkin)**:
```
Cenário 1 — Coleta de métricas unificada
Dado que um novo microsserviço é deployado
Quando o sistema inicializa
Então deve expor métricas no formato Prometheus automaticamente
E deve registrar no service discovery
E deve aparecer no Grafana dentro de 60 segundos

Cenário 2 — Detecção de anomalia com ML
Dado que o padrão de requisições do último mês está no modelo
Quando o tráfego atual desvia 3 desvios padrão do esperado
Então o sistema deve gerar alerta preditivo nível 1
E deve sugerir ação corretiva baseada em histórico
E deve atualizar modelo com novo padrão

Cenário 3 — Dashboard de autonomia
Dado que o PO acessa o dashboard de autonomia
Quando visualiza a página principal
Então deve ver uptime dos últimos 30 dias
E taxa de auto-resolução de incidentes
E métricas de saúde agregadas por serviço
E previsão de capacidade para próxima semana
```

**DEPENDÊNCIAS**: Nenhuma (fundação)

**MOCKUP/REFERÊNCIA**: `/wireframes/monitoring-dashboard.png`

**NOTAS TÉCNICAS**:
- Usar Prometheus + Grafana + Cortex
- Implementar modelo isolation forest para detecção de anomalias
- Armazenar métricas por 13 meses para análise sazonal

#### História 4: Sistema de Deploy Autônomo com Rollback Automático
**HISTÓRIA**:
Como DevOps Engineer,
Quero que deploys ocorram automaticamente com validação e rollback seguro,
Para que possamos entregar valor continuamente sem risco de downtime.

**CRITÉRIOS DE ACEITE (Gherkin)**:
```
Cenário 1 — Pipeline de deploy contínuo
Dado que um PR é mergeado na branch main
Quando o código passa em todos os testes automatizados
Então o sistema deve construir imagem Docker automaticamente
E deve executar testes de integração em ambiente staging
E se aprovado, deve promover para produção gradualmente (5%, 25%, 50%, 100%)

Cenário 2 — Validação pós-deploy automática
Dado que um novo deploy atingiu 100% do tráfego
Quando o sistema monitora por 10 minutos
Então deve verificar taxa de erro (< 0.1%)
E deve verificar latência P95 (< 200ms)
E deve comparar métricas com baseline pré-deploy
E se degradação detectada, deve iniciar rollback automático

Cenário 3 — Rollback automático por degradação
Dado que a taxa de erro aumentou para > 1% após deploy
Quando o sistema detecta por 2 minutos consecutivos
Então deve reverter para versão anterior automaticamente
E deve notificar equipe com análise root cause
E deve bloquear novos deploys até causa ser identificada
```

**DEPENDÊNCIAS**:
- História 3 (Monitoramento)
- História 7 (Testes Automatizados)

**MOCKUP/REFERÊNCIA**: N/A

**NOTAS TÉCNICAS**:
- Implementar feature flags para deploy gradual
- Usar canary analysis com Istio
- Manter últimas 3 versões sempre disponíveis para rollback

#### História 5: Orquestração de Containers com Auto-otimização
**HISTÓRIA**:
Como SRE,
Quero que o sistema otimize automaticamente alocação de recursos,
Para que custo de infraestrutura seja minimizado sem comprometer performance.

**CRITÉRIOS DE ACEITE (Gherkin)**:
```
Cenário 1 — Otimização de recursos por uso real
Dado que um pod está usando apenas 40% da CPU alocada por 24h
Quando o VPA (Vertical Pod Autoscaler) analisa métricas
Então deve sugerir redução de request/limit em 25%
E deve aplicar após aprovação automática (se dentro de limites seguros)
E deve monitorar performance por 2h após mudança

Cenário 2 — Bin packing automático
Dado que há nós no cluster com < 30% de utilização
Quando o scheduler executa otimização noturna
Então deve consolidar pods para menor número de nós possível
E deve garantir que pods críticos não fiquem no mesmo nó
E deve gerar relatório de economia estimada

Cenário 3 — Escalonamento de cluster
Dado que utilização média do cluster excede 80% por 1h
Quando o cluster autoscaler verifica capacidade
Então deve adicionar 1 nó automaticamente ao cluster
E deve redistribuir carga após nó estar saudável
E deve remover nó quando utilização cair abaixo de 40% por 4h
```

**DEPENDÊNCIAS**:
- História 2 (Escalonamento Horizontal)
- História 8 (Gestão de Custos)

**MOCKUP/REFERÊNCIA**: N/A

**NOTAS TÉCNICAS**:
- Usar Kubernetes VPA com mode=Auto
- Implementar desfragmentação semanal de cluster
- Configurar pod anti-affinity para alta disponibilidade

#### História 6: Service Mesh com Resiliência Automática
**HISTÓRIA**:
Como Backend Developer,
Quero que comunicações entre serviços tenham resiliência automática,
Para que falhas parciais não causem cascata em todo sistema.

**CRITÉRIOS DE ACEITE (Gherkin)**:
```
Cenário 1 — Circuit breaker automático
Dado que um serviço downstream está com 50% de taxa de erro
Quando um serviço upstream faz requisição
Então o circuit breaker deve abrir após 5 falhas consecutivas
E deve rejeitar requisições por 30 segundos (fast fail)
E deve tentar meio-aberto após período com 1 requisição teste

Cenário 2 — Retry com backoff exponencial
Dado que uma requisição falha com erro transitório (HTTP 503)
Quando o service mesh intercepta a falha
Então deve retentar após 100ms, 200ms, 400ms (exponencial)
E deve parar após 3 tentativas
E deve propagar erro original se todas falharem

Cenário 3 — Timeout dinâmico por percentil
Dado que o P99 de um serviço é 800ms
Quando o service mesh configura timeout automático
Então deve definir timeout como P99 + 20% (960ms)
E deve ajustar semanalmente baseado em métricas reais
E deve logar timeout como alerta se > 2 segundos
```

**DEPENDÊNCIAS**:
- História 3 (Monitoramento)
- História 4 (Deploy Autônomo)

**MOCKUP/REFERÊNCIA**: N/A

**NOTAS TÉCNICAS**:
- Implementar com Istio ou Linkerd
- Configurar fault injection em staging para testar resiliência
- Implementar dark traffic para testar novos serviços

#### História 7: Testes Automatizados com Geração de Casos Autônoma
**HISTÓRIA**:
Como QA Engineer,
Quero que o sistema gere e execute testes automaticamente baseado em mudanças de código,
Para que cobertura permaneça alta sem esforço manual constante.

**CRITÉRIOS DE ACEITE (Gherkin)**:
```
Cenário 1 — Geração de testes unitários
Dado que um desenvolvedor commitou nova função
Quando o sistema analisa a assinatura e complexidade ciclomática
Então deve gerar 3-5 testes unitários cobrando caminhos básicos
E deve executar imediatamente no pipeline
E deve sugerar edge cases baseado em tipos de parâmetros

Cenário 2 — Detecção de regressão comportamental
Dado que um teste que passava agora falha
Quando o sistema analisa diff do código
Então deve identificar qual mudança causou regressão
E deve sugerar correção ou atualização de teste
E deve notificar autor do commit com contexto específico

Cenário 3 — Testes de integração autogerados
Dado que uma nova API REST é definida no OpenAPI spec
Quando o sistema detecta novos endpoints
Então deve gerar testes de integração cobrando:
  - Todos os status codes definidos
  - Validação de schema de resposta
  - Testes de carga básica (10 req/seg)
  - Testes de segurança (SQLi, XSS básico)
```

**DEPENDÊNCIAS**: Nenhuma (independente)

**MOCKUP/REFERÊNCIA**: N/A

**NOTAS TÉCNICAS**:
- Usar ferramentas como EvoSuite para Java, Pynguin para Python
- Integrar com SonarQube para análise estática
- Implementar mutation testing periódico

#### História 8: Sistema de Gestão de Custos Autônoma
**HISTÓRIA**:
Como Product Manager,
Quero que o sistema otimize custos de infraestrutura automaticamente,
Para que possamos maximizar ROI sem intervenção manual.

**CRITÉRIOS DE ACEITE (Gherkin)**:
```
Cenário 1 — Detecção de recursos ociosos
Dado que um recurso cloud tem utilização < 10% por 7 dias
Quando o sistema analisa custo vs utilização
Então deve sugerir downgrade ou término do recurso
E se aprovado automaticamente (baixo risco), deve executar
E deve monitorar por 48h para garantir não afetar performance

Cenário 2 — Otimização de instâncias reservadas
Dado que o sistema roda em cloud provider com instâncias reservadas
Quando analisa padrão de uso histórico
Então deve sugerir compra de reservas para instâncias estáveis
E deve calcular ROI esperado
E deve executar compra se ROI > 40% em 1 ano

Cenário 3 — Alertas de custo anômalo
Dado que custo diário aumentou 30% vs média semanal
Quando o sistema detecta anomalia de custo
Então deve identificar recurso responsável pelo aumento
E deve sugerir ação corretiva (scale-in, tipo diferente de instância)
E deve notificar equipe financeira se aumento > $500/dia
```

# DIYAPP Evolution - V12 Core - Arquitetura Técnica

## Arquitetura Decision Record (ADR-001): Stack Técnica V12

**Data:** 2024-01-15
**Status:** Aceita
**Autores:** Tech Lead

### CONTEXTO:
A V11 do DIYAPP atingiu maturidade funcional mas apresenta desafios de:
1. Manutenção complexa devido a acoplamento entre componentes
2. Dificuldade em escalar funcionalidades de IA de forma isolada
3. Monitoramento insuficiente para operação 100% autônoma
4. Deploy manual propenso a erros humanos

### DECISÃO:
Adotar arquitetura de microserviços com frontend unificado, utilizando:
- **Frontend:** React 18 + TypeScript + Vite
- **Backend Services:** Node.js 20 + Express + TypeScript
- **Banco de Dados:** PostgreSQL (dados estruturados) + Redis (cache/sessões)
- **Message Queue:** RabbitMQ para comunicação assíncrona entre serviços
- **Containerização:** Docker + Docker Compose para desenvolvimento
- **Orquestração:** Kubernetes para produção
- **Monitoramento:** Prometheus + Grafana + ELK Stack

### OPÇÕES CONSIDERADAS:
- **Opção A (Monolito Aprimorado):** Manter arquitetura monolítica com melhorias incrementais
  - Prós: Menor complexidade inicial, deploy simplificado
  - Contras: Limita escalabilidade, acoplamento persistente, dificulta refatoração paralela
- **Opção B (Microserviços Leves):** Arquitetura proposta acima
  - Prós: Escalabilidade independente, isolamento de falhas, facilita Modo Hive
  - Contras: Complexidade operacional maior, necessidade de orquestração
- **Opção escolhida:** B - Justificativa: Alinha com objetivo de 100% autonomia e permite refatoração paralela via Modo Hive

### CONSEQUÊNCIAS:
**Positivas:**
- Serviços podem ser desenvolvidos/refatorados independentemente
- Escalabilidade granular por componente
- Isolamento de falhas entre serviços
- Facilita implementação de estratégias de resiliência

**Negativas:**
- Maior complexidade de deploy e monitoramento
- Overhead de comunicação entre serviços
- Necessidade de infraestrutura mais robusta

**Riscos:**
- Latência em chamadas entre serviços (mitigar com cache Redis)
- Complexidade de debug distribuído (mitigar com tracing distribuído)
- Gerenciamento de banco de dados distribuído (mitigar com API Gateway)

```typescript
import express from 'express';
import cors from 'cors';
import helmet from 'helmet';
import { createLogger, format, transports } from 'winston';
import { errorHandler } from '../shared/middleware/error-handler';
import { requestLogger } from '../shared/middleware/request-logger';
import { healthRouter } from './routes/health';
import { tasksRouter } from './routes/tasks';
import { config } from '../shared/config';

const logger = createLogger({
  level: config.logLevel,
  format: format.combine(
    format.timestamp(),
    format.errors({ stack: true }),
    format.json()
  ),
  transports: [
    new transports.Console(),
    new transports.File({ filename: 'logs/error.log', level: 'error' }),
    new transports.File({ filename: 'logs/combined.log' })
  ]
});

const app = express();

// Middlewares
app.use(helmet());
app.use(cors());
app.use(express.json());
app.use(requestLogger(logger));

// Routes
app.use('/health', healthRouter);
app.use('/api/v1/tasks', tasksRouter);

// Error handling
app.use(errorHandler);

const PORT = config.port || 3000;

app.listen(PORT, () => {
  logger.info(`Task Service started on port ${PORT}`);
});

// Graceful shutdown
process.on('SIGTERM', () => {
  logger.info('SIGTERM received, shutting down gracefully');
  process.exit(0);
});

process.on('SIGINT', () => {
  logger.info('SIGINT received, shutting down gracefully');
  process.exit(0);
});

export { app, logger };
```

```javascript
const urlsToCache = [
  '/',
  '/index.html',
  '/src/styles/design-tokens.css',
  '/src/styles/global.css',
  '/assets/icons/favicon.svg',
  '/assets/icons/apple-touch-icon.png'

// Install event
self.addEventListener('install', event => {
  event.waitUntil(
    caches.open(CACHE_NAME)
      .then(cache => {
        console.log('Opened cache');
        return cache.addAll(urlsToCache);
      })
  );
});

// Activate event
self.addEventListener('activate', event => {
  event.waitUntil(
    caches.keys().then(cacheNames => {
      return Promise.all(
        cacheNames.map(cacheName => {
          if (cacheName !== CACHE_NAME) {
            console.log('Deleting old cache:', cacheName);
            return caches.delete(cacheName);
          }
        })
      );
    })
  );
});

// Fetch event with network-first strategy
self.addEventListener('fetch', event => {
  event.respondWith(
    fetch(event.request)
      .then(response => {
        // Check if we received a valid response
        if (!response || response.status !== 200 || response.type !== 'basic') {
          return response;
        }

        // Clone the response
        const responseToCache = response.clone();

        caches.open(CACHE_NAME)
          .then(cache => {
            cache.put(event.request, responseToCache);
          });

        return response;
      })
      .catch(() => {
        // Network failed, try cache
        return caches.match(event.request)
          .then(response => {
            if (response) {
              return response;
            }
            // Return offline page if available
            return caches.match('/offline.html');
          });
      })
  );
});
```

```javascript
const { Pool } = require('pg');
const logger = require('../utils/logger');

class Database {
  constructor() {
    this.pool = null;
    this.isInitialized = false;
  }

  async initialize() {
    if (this.isInitialized) return;
    
    this.pool = new Pool({
      host: process.env.DB_HOST || 'localhost',
      port: process.env.DB_PORT || 5432,
      database: process.env.DB_NAME || 'diyapp',
      user: process.env.DB_USER || 'postgres',
      password: process.env.DB_PASSWORD,
      max: 20,
      idleTimeoutMillis: 30000,
      connectionTimeoutMillis: 2000,
    });

    // Test connection
    try {
      const client = await this.pool.connect();
      logger.info('Database connected successfully');
      client.release();
      this.isInitialized = true;
    } catch (error) {
      logger.error('Database connection failed:', error);
      throw error;
    }

    // Error handling
    this.pool.on('error', (err) => {
      logger.error('Unexpected database error:', err);
      this.isInitialized = false;
    });
  }

  async query(text, params) {
    if (!this.isInitialized) {
      throw new Error('Database not initialized');
    }

    const startTime = Date.now();
    try {
      const result = await this.pool.query(text, params);
      const duration = Date.now() - startTime;
      
      // Log slow queries
      if (duration > 1000) {
        logger.warn({
          message: 'Slow query detected',
          query: text,
          params: params,
          duration_ms: duration,
        });
      }
      
      return result;
    } catch (error) {
      logger.error({
        message: 'Database query failed',
        query: text,
        params: params,
        error: error.message,
      });
      throw error;
    }
  }

  async transaction(callback) {
    const client = await this.pool.connect();
    
    try {
      await client.query('BEGIN');
      const result = await callback(client);
      await client.query('COMMIT');
      return result;
    } catch (error) {
      await client.query('ROLLBACK');
      throw error;
    } finally {
      client.release();
    }
  }

  async close() {
    if (this.pool) {
      await this.pool.end();
      logger.info('Database connection closed');
    }
  }
}

module.exports = new Database();
```

```javascript
const { createClient } = require('redis');

class RedisClient {
  constructor() {
    this.client = null;
    this.isConnected = false;

  async initialize() {
    if (this.isConnected) return;

    this.client = createClient({
      url: process.env.REDIS_URL || 'redis://localhost:6379',
      socket: {
        reconnectStrategy: (retries) => {
          if (retries > 10) {
            logger.error('Redis reconnection failed after 10 attempts');
            return new Error('Max reconnection attempts reached');
          }
          return Math.min(retries * 100, 3000);
        },
      },
    });

    this.client.on('error', (err) => {
      logger.error('Redis error:', err);
      this.isConnected = false;
    });

    this.client.on('connect', () => {
      logger.info('Redis connected successfully');
      this.isConnected = true;
    });

    this.client.on('reconnecting', () => {
      logger.info('Redis reconnecting...');
    });

    await this.client.connect();
  }

  async get(key) {
    if (!this.isConnected) {
      throw new Error('Redis not connected');
    }

    try {
      return await this.client.get(key);
    } catch (error) {
      logger.error('Redis get failed:', error);
      throw error;
    }
  }

  async set(key, value, ttl = 3600) {
    if (!this.isConnected) {
      throw new Error('Redis not connected');
    }

    try {
      await this.client.set(key, value, { EX: ttl });
    } catch (error) {
```

### Documento de Threat Modeling

### Análise STRIDE:

#### S - Spoofing (Falsificação)
- **Ameaça:** Ataque de força bruta em endpoints de autenticação
- **Ameaça:** Token JWT roubado/reutilizado
- **Ameaça:** Credenciais de serviço comprometidas
- **Controles:** MFA obrigatório, rate limiting, tokens de curta duração, rotação de secrets

#### T - Tampering (Manipulação)
- **Ameaça:** Manipulação de dados em trânsito (MITM)
- **Ameaça:** Injeção de SQL/NoSQL
- **Ameaça:** Prompt injection em LLMs
- **Controles:** TLS 1.3, validação de input, prepared statements, sanitização de prompts

#### R - Repudiation (Repúdio)
- **Ameaça:** Negação de ações administrativas
- **Ameaça:** Logs manipulados ou apagados
- **Controles:** Logs imutáveis (WAL), auditoria de todas as operações críticas, assinatura digital de logs

#### I - Information Disclosure (Divulgação)
- **Ameaça:** Vazamento de dados sensíveis via logs
- **Ameaça:** Exposição de dados em respostas de erro
- **Ameaça:** Data exfiltration via LLM
- **Controles:** Criptografia em repouso, mascaramento de PII em logs, sanitização de outputs LLM

#### D - Denial of Service (Negação)

- **Controles:** Rate limiting por IP/usuario, circuit breakers, quotas de uso, timeout em processamento LLM

#### E - Elevation of Privilege (Elevação)
- **Ameaça:** IDOR (Insecure Direct Object Reference)
- **Ameaça:** Bypass de autorização
- **Ameaça:** Prompt injection para execução de comandos
- **Controles:** RBAC granular, verificação de permissão por recurso, princípio do menor privilégio, sandboxing de LLM

### Riscos Aceitos Conscientemente:

```javascript
const crypto = require('crypto');
const jwt = require('jsonwebtoken');
const { createHash } = require('crypto');

class OAuth2Provider {
    constructor() {
        this.clients = new Map();
        this.authCodes = new Map();
        this.accessTokens = new Map();
        this.refreshTokens = new Map();
    }

    // Registro de cliente OAuth2
    registerClient(clientId, clientSecret, redirectUris, scopes) {
        const client = {
            id: clientId,
            secret: this.hashSecret(clientSecret),
            redirectUris,
            scopes,
            createdAt: new Date()
        };
        this.clients.set(clientId, client);
        return client;
    }

    // Geração de authorization code
    generateAuthCode(clientId, userId, scopes) {
        const code = crypto.randomBytes(32).toString('hex');
        const codeData = {
            clientId,
            userId,
            scopes,
            expiresAt: Date.now() + (10 * 60 * 1000), // 10 minutos
            used: false
        };
        this.authCodes.set(code, codeData);
        return code;
    }

    // Troca de code por access token
    async exchangeCodeForToken(code, clientId, clientSecret) {
        const codeData = this.authCodes.get(code);
        
        if (!codeData) {
            throw new Error('Invalid authorization code');
        }
        
        if (codeData.used) {
            throw new Error('Authorization code already used');
        }
        
        if (codeData.expiresAt < Date.now()) {
            throw new Error('Authorization code expired');
        }
        
        if (codeData.clientId !== clientId) {
            throw new Error('Client mismatch');
        }
        
        const client = this.clients.get(clientId);
        if (!client || !this.verifySecret(clientSecret, client.secret)) {
            throw new Error('Invalid client credentials');
        }
        
        // Marca code como usado
        codeData.used = true;
        this.authCodes.set(code, codeData);
        
        // Gera tokens
        return this.generateTokens(clientId, codeData.userId, codeData.scopes);
    }

    // Geração de tokens JWT
    async generateTokens(clientId, userId, scopes) {
        const accessToken = jwt.sign(
            {
                sub: userId,
                client_id: clientId,
                scope: scopes.join(' '),
                type: 'access'
            },
            process.env.JWT_SECRET,
            { 
                expiresIn: '15m',
                algorithm: 'HS256'
            }
        );
        
        const refreshToken = crypto.randomBytes(64).toString('hex');
        const refreshTokenHash = this.hashSecret(refreshToken);
        
        this.refreshTokens.set(refreshTokenHash, {
            userId,
            clientId,
            scopes,
            expiresAt: Date.now() + (7 * 24 * 60 * 60 * 1000) // 7 dias
        });
        
        return {
            access_token: accessToken,
            refresh_token: refreshToken,
            token_type: 'Bearer',
            expires_in: 900,
            scope: scopes.join(' ')
        };
    }

    // Refresh token
    async refreshAccessToken(refreshToken, clientId) {
        const refreshTokenHash = this.hashSecret(refreshToken);
        const tokenData = this.refreshTokens.get(refreshTokenHash);
        
        if (!tokenData) {
            throw new Error('Invalid refresh token');
        }
        
        if (tokenData.expiresAt < Date.now()) {
            this.refreshTokens.delete(refreshTokenHash);
            throw new Error('Refresh token expired');
        }
        
        if (tokenData.clientId !== clientId) {
            throw new Error('Client mismatch');
        }
        
        // Invalida refresh token antigo
        this.refreshTokens.delete(refreshTokenHash);
        
        // Gera novos tokens
        return this.generateTokens(clientId, tokenData.userId, tokenData.scopes);
    }

    // Revogação de tokens
    revokeTokens(refreshToken) {
        const refreshTokenHash = this.hashSecret(refreshToken);
        this.refreshTokens.delete(refreshTokenHash);
    }

    // Utilitários de hash
    hashSecret(secret) {
        return createHash('sha256').update(secret).digest('hex');
    }

    verifySecret(inputSecret, storedHash) {
        return this.hashSecret(inputSecret) === storedHash;
    }
}

module.exports = OAuth2Provider;
```

**src/auth/jwt.js:**
```javascript
const jwt = require('jsonwebtoken');
const jwksClient = require('jwks-rsa');

class JWTManager {
    constructor() {
        this.blacklist = new Set();
        
        // Configuração para JWKS (para uso com RSA)
        this.jwksClient = jwksClient({
            jwksUri: process.env.JWKS_URI || 'https://auth.diyapp.com/.well-known/jwks.json',
            cache: true,
            rateLimit: true,
            jwksRequestsPerMinute: 10
        });
    }

    // Verificação de token JWT
    async verifyToken(token, options = {}) {
        if (this.blacklist.has(this.getTokenId(token))) {
            throw new Error('Token revoked');
        }

        const verifyOptions = {
            algorithms: ['HS256', 'RS256'],
            issuer: process.env.JWT_ISSUER || 'diyapp-v12',
            audience: process.env.JWT_AUDIENCE || 'diyapp-api',
            ...options
        };

        try {
            // Para tokens HS256 (simétrico)
            if (process.env.JWT_ALGORITHM === 'HS256') {
                return jwt.verify(token, process.env.JWT_SECRET, verifyOptions);
            }
            
            // Para tokens RS256 (assimétrico)
            const decoded = jwt.decode(token, { complete: true });
            if (!decoded) {
                throw new Error('Invalid token');
            }
            
            const key = await this.jwksClient.getSigningKey(decoded.header.kid);
            const signingKey = key.getPublicKey();
            
            return jwt.verify(token, signingKey, verifyOptions);
        } catch (error) {
            if (error.name === 'TokenExpiredError') {
                throw new Error('Token expired');
            }
            if (error.name === 'JsonWebTokenError') {
                throw new Error('Invalid token');
            }
            throw error;
        }
    }

    // Decodificação sem verificação (apenas para logging)
    decodeToken(token) {
        return jwt.decode(token, { complete: true });
    }

    // Adiciona token à blacklist
    blacklistToken(token) {
        const tokenId = this.getTokenId(token);
        const decoded = this.decodeToken(token);
        
        if (decoded && decoded.payload.exp) {
            const expiresIn = decoded.payload.exp * 1000 - Date.now();
            if (expiresIn > 0) {
                this.blacklist.add(tokenId);
                
                // Remove da blacklist após expiração
                setTimeout(() => {
                    this.blacklist.delete(tokenId);
                }, expiresIn);
            }
        }
    }

    // Gera ID único para o token
    getTokenId(token) {
        const decoded = this.decodeToken(token);
        if (!decoded) {
            return crypto.createHash('sha256').update(token).digest('hex');
        }
        
        return `${decoded.payload.jti || decoded.payload.sub}_${decoded.payload.iat}`;
    }

    // Validação de escopos
    validateScopes(tokenScopes, requiredScopes) {
        if (!tokenScopes || !requiredScopes) return false;
        
        const tokenScopeArray = tokenScopes.split(' ');
        const requiredScopeArray = Array.isArray(requiredScopes) 
            ? requiredScopes 
            : requiredScopes.split(' ');
        
        return requiredScopeArray.every(scope => tokenScopeArray.includes(scope));
    }
}

module.exports = JWTManager;
```
