# ADR-001: DIYAPP V12 Core Architecture

**Data:** 2024-03-15
**Status:** Aceita
**Autores:** Tech Lead + Especialista Infra + Especialista Backend

## CONTEXTO:
DIYAPP evoluiu para V12 com requisitos de:
1. 100% estabilidade autônoma (zero downtime deployments)
2. Modo Hive para refatoração paralela de componentes
3. Performance P95 < 200ms para todas APIs críticas
4. Escalabilidade horizontal automática
5. Observabilidade completa em tempo real

Forças em jogo:
- Necessidade de deploy contínuo sem interromper funcionalidades existentes
- Equipes distribuídas trabalhando em paralelo no mesmo sistema
- Mix de componentes legados (V11) e novos (V12)
- Crítico: manutenção da integridade de dados durante transições

## DECISÃO:
Adotar arquitetura de microsserviços com **Backend-for-Frontend (BFF) pattern** e **Service Mesh** para orquestração.

**Stack Principal:**
- **Runtime:** Node.js 20 LTS (ESM modules)
- **Frontend:** React 18 + TypeScript + Vite
- **API Gateway:** Kong (com plugins de rate limiting, auth, logging)
- **Service Mesh:** Linkerd 2.14 (para comunicação entre serviços)
- **Banco de Dados:**
  - PostgreSQL 16 (dados transacionais)
  - Redis 7 (cache + sessões + filas)
  - MongoDB 7 (dados não relacionais/documents)
- **Mensageria:** RabbitMQ 3.12 (event-driven architecture)
- **Monitoramento:** Prometheus + Grafana + Jaeger (tracing distribuído)

**Padrões de API:**
- RESTful para CRUD operations (OpenAPI 3.0 specification)
- GraphQL para queries complexas (Apollo Server)
- gRPC para comunicação interna entre serviços
- WebSocket para real-time updates

**Estratégia de Estado:**
- **Stateless services:** Autenticação via JWT tokens (refresh rotation)
- **Stateful services:** Session stickiness via Redis Cluster
- **Cache distribuído:** Redis com TTL automático e invalidation pattern
- **Uploads:** S3-compatible storage (MinIO para dev, AWS S3 para prod)

## OPÇÕES CONSIDERADAS:

### Opção A: Monolito Modular (Laravel + Livewire)
- **Prós:** 
  - Desenvolvimento rápido
  - Debugging simplificado
  - Transações ACID fáceis
- **Contras:**
  - Acoplamento alto dificulta refatoração paralela
  - Scaling vertical limitado
  - Deploy all-or-nothing
- **Rejeitada:** Incompatível com Modo Hive e 100% uptime

### Opção B: Microsserviços Puros (Go + gRPC)
- **Prós:**
  - Desacoplamento máximo
  - Performance excelente
  - Polyglot persistence
- **Contras:**
  - Complexidade operacional alta
  - Latência de rede entre serviços
  - Debugging distribuído difícil
- **Rejeitada:** Over-engineering para nosso tamanho atual

### Opção C: BFF + Service Mesh (Escolhida)
- **Prós:**
  - Frontends independentes podem evoluir separadamente
  - Service Mesh gerencia comunicação, retry, circuit breaking
  - Modo Hive natural: cada BFF é uma "colmeia" independente
  - Deploy gradual (canary releases via Linkerd)
- **Contras:**
  - Duplicação de lógica em BFFs
  - Necessidade de service mesh (complexidade adicional)
- **Escolhida:** Balance ideal entre autonomia e complexidade

## CONSEQUÊNCIAS:

### Positivas:
1. **Refatoração Paralela:** Equipes podem trabalhar em BFFs independentes
2. **Zero Downtime:** Linkerd permite canary deployments e traffic shifting
3. **Observabilidade:** Tracing distribuído identifica bottlenecks rapidamente
4. **Resiliência:** Circuit breakers previnem cascading failures
5. **Performance:** Cache layers + CDN garantem P95 < 200ms

### Negativas:
1. **Complexidade Operacional:** 5x mais componentes para monitorar
2. **Latência de Rede:** Comunicação entre serviços adiciona 10-50ms
3. **Custo Infra:** Service mesh + múltiplos bancos aumentam custo em ~40%
4. **Debugging:** Erros distribuídos requerem Jaeger + logs centralizados

### Riscos e Mitigações:
| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| Service Mesh overhead | Alta | Médio | Benchmarks regulares, otimização de resource limits |
| Data consistency entre serviços | Média | Alto | Event sourcing + sagas pattern, compensação transactions |
| Vendor lock-in (Linkerd) | Baixa | Médio | Abstraction layer, possível migração para Istio |
| Cold start em serverless | Alta | Baixo | Provisioned concurrency, warm-up scripts |

```yaml
name: Deploy V12 Service
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
        
      - name: Lint
        run: npm run lint
        
      - name: Type check
        run: npm run type-check
        
      - name: Test with coverage
        run: npm test -- --coverage
        env:
          NODE_ENV: test
          
      - name: Upload coverage
        uses: codecov/codecov-action@v3

  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Build Docker image
        run: |
          docker build -t diyapp/${{ github.event.repository.name }}:${{ github.sha }} .
          docker tag diyapp/${{ github.event.repository.name }}:${{ github.sha }} diyapp/${{ github.event.repository.name }}:latest
          
      - name: Scan for vulnerabilities
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'diyapp/${{ github.event.repository.name }}:${{ github.sha }}'
          format: 'sarif'
          output: 'trivy-results.sarif'
          
      - name: Push to registry
        if: github.ref == 'refs/heads/main'
        run: |
          echo "${{ secrets.DOCKER_PASSWORD }}" | docker login -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
          docker push diyapp/${{ github.event.repository.name }}:${{ github.sha }}
          docker push diyapp/${{ github.event.repository.name }}:latest

  deploy:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging (canary 10%)
        uses: azure/k8s-deploy@v4
        with:
          namespace: diyapp-staging
          manifests: k8s/manifests/
          images: diyapp/${{ github.event.repository.name }}:${{ github.sha }}
          traffic-split-method: smi
          percentage: 10
          
      - name: Run integration tests
        run: npm run test:e2e -- --env=staging
        
      - name: Promote to 100% (if tests pass)
        if: success()
        uses: azure/k8s-deploy@v4
        with:
          namespace: diyapp-production
          manifests: k8s/manifests/
          images: diyapp/${{ github.event.repository.name }}:${{ github.sha }}
          traffic-split-method: smi
          percentage: 100
```

## PADRÕES DE COMUNICAÇÃO ENTRE SERVIÇOS:
```typescript
// packages/@diyapp/commons/src/communication.ts
export interface ServiceMessage<T = any> {
  id: string;
  type: string;
  timestamp: number;
  source: string;
  destination: string;
  payload: T;
  correlationId?: string;
  metadata?: Record<string, any>;
}

export class ServiceClient {
  private constructor(private serviceName: string) {}
  
  async call<T>(service: string, endpoint: string, data: any): Promise<T> {
    // Usa gRPC para serviços internos, REST para externos
    const protocol = this.isInternalService(service) ? 'grpc' : 'http';
    
    return await this.withRetry(async () => {
      return await this.makeRequest(protocol, service, endpoint, data);
    }, {
      maxRetries: 3,
      backoff: 'exponential'
    });
  }
  
  private async makeRequest(protocol: string, service: string, endpoint: string, data: any) {
    // Implementação via Linkerd mesh
    const url = this.buildUrl(protocol, service, endpoint);
    const headers = this.getHeaders();
    
    const response = await fetch(url, {
      method: 'POST',
      headers,
      body: JSON.stringify(data)
    });
    
    if (!response.ok) {
      throw new ServiceError(response.status, await response.text());
    }
    
    return await response.json();
  }
}
```

    origin: process.env.ALLOWED_ORIGINS ? process.env.ALLOWED_ORIGINS.split(',') : '*',
    credentials: true
}));

app.use(express.json({ limit: '10mb' }));
app.use(express.urlencoded({ extended: true }));

// Request logging middleware
app.use((req, res, next) => {
    const correlationId = req.headers['x-correlation-id'] || require('uuid').v4();
    req.correlationId = correlationId;
    
    const start = Date.now();
    
    res.on('finish', () => {
        const duration = Date.now() - start;
        
        logger.info({
            correlation_id: correlationId,
            method: req.method,
            url: req.url,
            status: res.statusCode,
            duration_ms: duration,
            user_agent: req.get('user-agent'),
            ip: req.ip
        });
        
        // Record metrics
        metrics.recordRequest(req.method, req.url, res.statusCode, duration);
    });
    
    next();
});

// Serve dashboard
app.get('/', (req, res) => {
    res.sendFile(path.join(__dirname, '../index.html'));
});

app.get('/health', (req, res) => {
    res.json({
        status: 'healthy',

        uptime: process.uptime(),
        version: 'v12.0.0',
        metrics: metrics.getMetrics()
    });
});
