# DIYAPP Evolution - V12 Core - Arquitetura de Microsserviços

## 1. ADR-001: Arquitetura de Microsserviços V12

**Data:** 2024-03-15
**Status:** Aceita
**Autores:** Tech Lead, Especialista Infra, Especialista Backend

### CONTEXTO:
A V11 do DIYAPP atingiu limites de escalabilidade devido à arquitetura monolítica. O sistema precisa:
- Suportar 10x mais usuários simultâneos
- Permitir deploy independente de funcionalidades
- Garantir 99.99% de disponibilidade
- Facilitar manutenção paralela por múltiplas squads

### DECISÃO:
Adotar arquitetura de microsserviços com os seguintes princípios:
1. **Domínios bem definidos** - Cada microsserviço encapsula um domínio de negócio
2. **Autonomia total** - Cada serviço tem seu próprio banco de dados
3. **Comunicação assíncrona por padrão** - Eventos via Apache Kafka
4. **Sincronização quando necessário** - gRPC para comunicações síncronas
5. **Observabilidade nativa** - Tracing distribuído, métricas e logs centralizados

### OPÇÕES CONSIDERADAS:

**Opção A: Monólito modularizado**
- Prós: Simples deploy, debugging fácil, transações ACID
- Contras: Escalabilidade limitada, acoplamento alto, deploy único
- **Rejeitada** - Não atende requisitos de escalabilidade

**Opção B: Microsserviços com REST/HTTP**
- Prós: Simples implementação, ampla adoção
- Contras: Latência alta, sem contrato forte, versionamento complexo
- **Rejeitada** - Falta de contrato forte e performance subótima

**Opção C: Microsserviços com gRPC + Eventos (ESCOLHIDA)**
- Prós: Contrato forte com Protobuf, alta performance, suporte a streaming
- Contras: Curva de aprendizado, tooling menos maduro
- **Justificativa:** Combina performance gRPC para comunicações síncronas com resiliência de eventos assíncronos

### CONSEQUÊNCIAS:
**Positivas:**
- Escalabilidade horizontal ilimitada
- Deploy independente por squad
- Resiliência a falhas isoladas
- Stack tecnológica heterogênea possível

**Negativas:**
- Complexidade operacional aumentada
- Transações distribuídas complexas
- Debugging mais difícil
- Overhead de rede

**Riscos:**
1. **Latência de rede** - Mitigação: Cache Redis, design assíncrono
2. **Consistência eventual** - Mitigação: Padrões SAGA, compensação
3. **Observabilidade** - Mitigação: Jaeger + Prometheus + Grafana

**Revisão:** 2024-06-15

### 2.2. Contratos gRPC (Protobuf)

```protobuf
// protos/user.proto
syntax = "proto3";

package diyapp.v12;

service UserService {
  rpc GetUser(GetUserRequest) returns (UserResponse);
  rpc CreateUser(CreateUserRequest) returns (UserResponse);
  rpc UpdateUser(UpdateUserRequest) returns (UserResponse);
}

message GetUserRequest {
  string user_id = 1;
}

message CreateUserRequest {
  string email = 1;
  string name = 2;
  string password_hash = 3;
}

message UpdateUserRequest {
  string user_id = 1;
  optional string name = 2;
  optional string avatar_url = 3;
}

message UserResponse {
  string user_id = 1;
  string email = 2;
  string name = 3;
  string avatar_url = 4;
  string created_at = 5;
  string updated_at = 6;
}

// protos/auth.proto
syntax = "proto3";

package diyapp.v12;

service AuthService {
  rpc Authenticate(AuthRequest) returns (AuthResponse);
  rpc ValidateToken(TokenRequest) returns (ValidationResponse);
  rpc RefreshToken(RefreshRequest) returns (AuthResponse);
}

message AuthRequest {
  string email = 1;
  string password = 2;
}

message AuthResponse {
  string access_token = 1;
  string refresh_token = 2;
  int32 expires_in = 3;
  string token_type = 4;
}

message TokenRequest {
  string token = 1;
}

message ValidationResponse {
  bool valid = 1;
  string user_id = 2;
  repeated string roles = 3;
}

message RefreshRequest {
  string refresh_token = 1;
}
```

## 1. Configuração CI/CD com GitOps (ArgoCD)

### 1.1 Pipeline CI/CD GitHub Actions

`.github/workflows/ci-pipeline.yml`:
```yaml
name: CI Pipeline - DIYAPP V12

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.9', '3.10', '3.11']

    steps:
    - uses: actions/checkout@v3
    
    - name: Set up Python ${{ matrix.python-version }}
      uses: actions/setup-python@v4
      with:
        python-version: ${{ matrix.python-version }}
    
    - name: Install dependencies
      run: |
        python -m pip install --upgrade pip
        pip install -r requirements.txt
        pip install pytest pytest-cov
    
    - name: Run tests with coverage
      run: |
        pytest tests/ --cov=src --cov-report=xml --cov-report=html
    
    - name: Upload coverage to Codecov
      uses: codecov/codecov-action@v3
      with:
        file: ./coverage.xml
        fail_ci_if_error: true

  security-scan:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    
    - name: Run Trivy vulnerability scanner
      uses: aquasecurity/trivy-action@master
      with:
        scan-type: 'fs'
        scan-ref: '.'
        format: 'sarif'
        output: 'trivy-results.sarif'
    
    - name: Upload Trivy scan results to GitHub Security tab
      uses: github/codeql-action/upload-sarif@v2
      with:
        sarif_file: 'trivy-results.sarif'

  build-and-push:
    needs: [test, security-scan]
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    
    permissions:
      contents: read
      packages: write
    
    steps:
    - uses: actions/checkout@v3
    
    - name: Log in to Container Registry
      uses: docker/login-action@v2
      with:
        registry: ${{ env.REGISTRY }}
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}
    
    - name: Extract metadata for Docker
      id: meta
      uses: docker/metadata-action@v4
      with:
        images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
        tags: |
          type=sha,prefix={{branch}}-
          type=ref,event=branch
          type=ref,event=pr
          type=semver,pattern={{version}}
          type=raw,value=latest,enable={{is_default_branch}}
    
    - name: Build and push Docker image
      uses: docker/build-push-action@v4
      with:
        context: .
        push: true
        tags: ${{ steps.meta.outputs.tags }}
        labels: ${{ steps.meta.outputs.labels }}
        cache-from: type=gha
        cache-to: type=gha,mode=max
```

      uses: actions/checkout@v3

        git config --local user.email "action@github.com"

        git commit -m "chore: update image tag to ${{ github.sha }}"
        git push origin main
    
    - name: Trigger ArgoCD sync

    toggleMode,

      {children}

const server = require('./server');
const config = require('./config');

async function startServer() {
  try {
    // Inicializar métricas
    metrics.initialize();
    
    // Iniciar servidor
    const app = await server();
    
    app.listen(config.port, () => {
      logger.info(`🚀 Servidor iniciado na porta ${config.port}`);
      logger.info(`📊 Métricas disponíveis em http://localhost:${config.port}/metrics`);
      logger.info(`📚 API Docs disponíveis em http://localhost:${config.port}/api-docs`);
      logger.info(`🎮 GraphQL Playground disponível em http://localhost:${config.port}/graphql`);
      logger.info(`👷 Queue Dashboard disponível em http://localhost:${config.port}/queues`);
      
      // Log de health check
      logger.info('✅ Sistema pronto para receber requisições');
    });
    
    // Graceful shutdown
    process.on('SIGTERM', () => {
      logger.info('🛑 SIGTERM recebido, iniciando graceful shutdown...');
      app.close(() => {
        logger.info('✅ Servidor finalizado com sucesso');
        process.exit(0);
      });
    });
    
    process.on('SIGINT', () => {
      logger.info('🛑 SIGINT recebido, iniciando graceful shutdown...');
      app.close(() => {
        logger.info('✅ Servidor finalizado com sucesso');
        process.exit(0);
      });
    });
    
    // Tratamento de erros não capturados
    process.on('unhandledRejection', (reason, promise) => {
      logger.error('❌ Unhandled Rejection:', { reason, promise });
      // Não encerrar o processo, apenas logar
    });
    
    process.on('uncaughtException', (error) => {
      logger.error('❌ Uncaught Exception:', error);
      // Não encerrar imediatamente, dar tempo para logs e métricas
      setTimeout(() => process.exit(1), 1000);
    });
    
  } catch (error) {
    logger.error('❌ Falha ao iniciar servidor:', error);
    process.exit(1);
  }
}

startServer();
