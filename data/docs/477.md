### [Sessão Paralela: Data Engineer]
Para otimizar as consultas de banco de dados, precisamos analisar cuidadosamente as consultas SQL existentes e identificar quaisquer problemas de desempenho, como operações de busca ineficientes, uso inadequado de índices ou junções complexas. Aqui está um guia sobre como podemos otimizar algumas consultas de banco de dados comuns focadas em PostgreSQL.

Se tivermos um conjunto de consultas SQL já escritas, nosso foco será otimizá-las. Presumo que estamos lidando com consultas complexas que precisam de melhorias, mas sem acesso ao código específico, criarei um exemplo genérico de como podemos proceder.

### Contexto
Digamos que temos uma tabela `orders` e outra tabela `customers` com uma consulta que faz uma junção entre essas duas para obter informações sobre pedidos e os respectivos clientes, mas está apresentando lentidão.

### Estrutura das Tabelas
Vamos supor as seguintes estruturas para simplificar:

```sql
CREATE TABLE customers (
    customer_id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);

CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    customer_id INT REFERENCES customers(customer_id),
    order_date TIMESTAMP,
    amount DECIMAL(10, 2)
);
```

### Consulta Original
A consulta existente pode ser algo assim:

```sql
SELECT c.customer_id, c.name, o.order_id, o.order_date, o.amount
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_date > '2023-01-01' AND c.name LIKE '%John%';
```

### Otimizações
1. **Índices Adequados**: Verifique se há índices nos campos usados para junção e na cláusula WHERE.
2. **Evitar LIKE ineficiente**: Se possível, substituir `LIKE '%John%'` por uma busca mais direta ou uma coluna indexada.
3. **Usar Campos Derivados**: Em consultas complexas, se o banco de dados permite, uso de colunas derivadas para evitar cálculos em tempo de execução.

### Passo a Passo da Otimização

#### 1. Criar Índices
Certifique-se de que as colunas usadas em junções e condições possuem índices:

```sql
CREATE INDEX idx_customer_name ON customers(name);
CREATE INDEX idx_order_date ON orders(order_date);
```

#### 2. Usar CTEs para Simplificar a Consulta
Common Table Expressions (CTEs) ajudam a simplificar consultas:

```sql
WITH FilteredCustomers AS (
    SELECT customer_id, name
    FROM customers
    WHERE name LIKE '%John%'
), RecentOrders AS (
    SELECT order_id, customer_id, order_date, amount
    FROM orders
    WHERE order_date > '2023-01-01'
)
SELECT fc.customer_id, fc.name, ro.order_id, ro.order_date, ro.amount
FROM FilteredCustomers fc
JOIN RecentOrders ro ON fc.customer_id = ro.customer_id;
```

```javascript
const { Given, When, Then } = require('cucumber');
const assert = require('assert');

Given('the user navigates to the login page', function() {
  // Simulate navigation to the login page
  this.page = 'login';
});

When('the user submits login credentials {string} and {string}', function(email, password) {
  // Simulate login action
  this.email = email;
  this.password = password;
  // Simulate authentication logic
  this.isAuthenticated = (email === 'user@example.com' && password === 'correctpassword');
});

Then('the user should be redirected to the dashboard', function() {
  assert.strictEqual(this.isAuthenticated, true, 'User is not authenticated');
  assert.strictEqual(this.page, 'login', 'User is not on the login page after authentication');
});

Then('an error message {string} should be displayed', function(errorMessage) {
  assert.strictEqual(this.isAuthenticated, false, 'User should not be authenticated');
  assert.strictEqual("Invalid credentials", errorMessage, "The expected error message was not displayed");
});
```
