Squad Autônoma Sênior
Diretrizes de governança, execução técnica e infraestrutura para squads 100% autônomas.
Princípios de operação
1
Autonomia com responsabilidade A squad opera de forma independente. Decisões técnicas, de produto e de design são tomadas internamente sem aprovação humana, exceto quando o Aprovador Automático escalona para nível humano por risco elevado.
2
Kanban como lei Toda e qualquer atividade da squad existe como um card no Kanban. Nada ocorre fora do board. O Kanban é a fonte única de verdade sobre o estado do trabalho.
3
Classificação antes de execução Nenhuma tarefa é executada sem ser classificada por complexidade, segurança e compliance. A classificação determina o caminho de aprovação obrigatório.
4
Decomposição como estratégia de risco Tarefas de alto risco não são bloqueadas — elas são quebradas em partes menores até atingirem nível baixo. Só quando a decomposição é impossível a aprovação humana é acionada.
5
GitHub como artefato final Todo output da squad termina em um commit, PR ou artefato no GitHub. Código, docs, configurações e decisões vivem no repositório. O Antigravity e o Claude Code são os executores; o GitHub é a memória.
6
Melhor LLM para cada tarefa O Especialista LLM avalia cada tipo de tarefa e recomenda o modelo mais adequado (Claude, Gemini, GPT, Mistral, etc.). Nenhum agente usa um modelo fixo — o modelo é determinado pela natureza da tarefa.
7
Aprendizado sistêmico Toda falha (bug ou incidente) é tratada como uma oportunidade de atualização da inteligência coletiva. O erro não apenas é corrigido no código, mas também "ensinado" ao agente responsável através da atualização de suas instruções.
Fluxo do Kanban autônomo
Backlog
Toda nova demanda, ideia ou bug entra aqui. Sem triagem, sem prioridade ainda.
→
Classificação
Aprovador Automático analisa complexidade + segurança + compliance e atribui nível.
→
Aprovação
Caminho determinado pelo nível. Baixo: auto. Médio/Alto: decomposição ou humano.
→
Em execução
Agente responsável executa. Antigravity orquestra; Claude Code implementa.
→
Done
PR mergeado, artefato entregue, resultado documentado pelo Tech Writer.
Lógica de aprovação por nível
Nível baixo
Aprovação automática
Tarefa isolada, sem toque em dados sensíveis, sem impacto regulatório. Execução imediata.
Nível médio
Decompor → reavaliar
Aprovador tenta quebrar em subtarefas de nível baixo. Se todas virarem baixo, executa. Se não, escala.
Nível alto
Decompor ou escalar humano
Mesma tentativa de decomposição. Se irreducível: notificação humana obrigatória com relatório de risco.

Meta-aprendizado
Cadência operacional — sprint de 2 semanas
Seg — Dia 1
Sprint Planning autônomo: PM + PO definem prioridades no Backlog. Aprovador classifica todos os cards. Squad Leader distribui por papel.
Diário
Daily async via Kanban: cada agente atualiza o status do seu card. Bloqueios são sinalizados automaticamente ao Squad Leader.
Sex — Dia 5
Mid-sprint check: Tech Lead revisa PRs abertos. QA valida o que foi entregue. SRE verifica alertas de confiabilidade.
Sex — Dia 10
Sprint Review + Retrospectiva: Demo do que foi entregue. AI Ops analisa custo e performance dos LLMs usados. Tech Writer publica documentação da sprint.

Círculo de Aprendizado Contínuo
DETECÇÃO (QA / SRE) └─ Bug crítico/alto ou Incidente Sev1/2 resolvido └─ ACIONAMENTO: Agente de Melhoria Contínua ├─ Analisa Causa Raiz: "Por que o agente falhou?" ├─ Gera PATCH DE INSTRUÇÃO (Novo checklist ou regra inviolável) └─ VALIDAÇÃO: Tech Leader aprova a mudança no HTML da Role └─ IMPLEMENTAÇÃO: O agente agora possui o novo conhecimento
O aprendizado é imortalizado no repositório. Uma vez que um erro ocorre, o sistema é vacinado contra a repetição desse padrão através da evolução do prompt de sistema do agente responsável.
Regra de ouro: nenhuma aprovação humana deve ser a primeira resposta a uma tarefa complexa. A decomposição é sempre tentada antes. A aprovação humana é o último recurso — não o caminho padrão.
